Abstract

This specification describes an object UAs can provide to extensions that includes metadata about the non-customized version of a new tab page.

1. Introduction

This section is non-normative.

A popular form of web extension replaces a UA’s new tab page (NTP) with some other custom functionality. By definition, this removes the content on the NTP. As a result, it is difficult for those extensions to fully match the theming a user has selected, or include useful content like bookmarks or recent tabs. This document proposes a new permission 1 that exposes a metadata object 2 that contains NTP theming 3 and content 4 information.

2. Definitions

Throughout this document, the term "original" refers to the state as it would be if no extension had replaced the NTP. Any theming, bookmarking, organiziation, etc. done by the user should be included when determining the "original" value. This means UAs MUST update any data to reflect any extension that augments (i.e. changes the visual style) the page without changing the content thereof. Any data content added to an otherwise "original" page DOES NOT need to be exposed.

2.1. NTPMetadata

dictionary NTPMetadata {
    NTPStyle style;
    FrozenArray<NTPItem> items;
};

NTPMetadata is a readonly object that an extension can use to learn about a user’s new tab page theming and contents.

  • style is an readonly object that provides information about the styling that would originally be used for the NTP
  • items is a FrozenArray of items that would originally be featured on the NTP. This would included the user’s favorite pages, bookmarks, reading list, etc. It is up to a UA what data they wish to include, with a preference towards inclusion.

    The ordering of items MUST reflect any user customized value (i.e. if the user manually arranged "item A" to be the first reading list entry, then it SHALL be the first item in items with where type is "readingListEntry"

    If the NTP was configured by the user to not show anything considered as a NTPItem, or if the UA decides there is no data to share, then items MUST be an empty array.

2.2. NTPItem

dictionary NTPItem {
    USVString favicon;
    USVString screenshot;
    DOMString href;
    DOMString previewSnippet;
    DOMString title;
    NTPItemType type;
};

NTPItem is a generic object that represents an individual entry in NTPMetadata.items.

  • favicon: A valid <bg-image> that points to the image that would originally be used as a favicon for this item on NTP. REQUIRED unless, `screenshot` is defined, in which case it MAY be set to "none"
  • screenshot: A valid <bg-image> that would originally be used as a screenshot for this item on NTP. REQUIRED unless, `favicon` is defined, in which case it MAY be set to "none"
  • title: A string title for the item.
  • href: A valid non-empty url the content for the item.
  • type: One of NTPItemType. Useful to differentiate between types of content used in some UAs NTP.
  • previewSnippet: A string representing any preview text used for cards on NTP. If no preview text is used, the value MUST be an empty string.

2.3. NTPStyle

NTPStyle is an object that exposes the styling that would be used for the NTP, if the extension was not being used. The properties SHOULD be limited to those customizable by user theming. 
dictionary NTPStyle {
    DOMString backgroundColor;
    DOMString backgroundImage = "none";
    DOMString backgroundPosition;
    DOMString backgroundRepeat;
    DOMString textColor;
    DOMString linkColor;
    DOMString itemHoverColor;
};
  • backgroundColor: A stringified <color>. Represents the original NTP background color.
  • backgroundImage: A stringified <image>. Represents the image originally used as a background on the NTP. Defaults to the keyword "none" unless otherwise defined.
  • backgroundPosition: A stringified <bg-position>. Represents any modification to the position of a background image used on the original NTP
  • backgroundRepeat: A stringified <repeat-style>. Represents any modification to the repetition of a background image used on the original NTP.
  • textColor: A stringified <color>. Represents the original NTP plain text color.
  • linkColor: A stringified <color>. Represents the color used for the text portion of links used in the original NTP. Pseudo class values SHOULD NOT be provided.
  • itemHoverColor: A stringifed <color>. Represents the color used for the backgroundColor on a NTPItem during interactive (e.g. :hover, :active) states on the original NTP.

should we add a shorthand `background` property for convenience?

2.4. NTPItemType

enum NTPItemType {
  "bookmark",
  "recent",
  "suggestion",
  "readingListEntry",
};
  • bookmark: Used for any bookmarks prominently featured on the original NTP.
  • recent: Used for any recent tabs that would be featured on the original NTP.
  • suggestion: Used for an item that is based on a UAs recommendation, rather than the user’s direct history or bookmark that would be featured on the original NTP.
  • readingListEntry: Used for any link that would be featured as an entry on a Reading List on the original NTP.

2.5. Extending properties

This document aims to cover any use case of UA or user customization of the NTP. However, there will inevitably be concepts or values missing. A UA MAY extend member defined by this document. When a UA extends NTPMetadata, NTPItem, or NTPStyle, they MUST abide by the following rules.

Any new key MUST be prefixed by the letter "x" (U+0078 LATIN SMALL LETTER X), followed by the a dash (U+002D HYPHEN-MINUS).

const {styles} = browser.ntpInfo
const logo = document.querySelector('#browserLogo');

logo.src = styles['x-ntp-browserLogo']

Additionally, in the specific case of NTPStyle properties, the same § 2.5 Extending properties rules MUST be followed for any custom environment variables (see § 3.2 in CSS)

main#customNewTabPage {
  fakeProperty: env('x-customValue')
}

3. Use

3.1. in Javascript

A NTPMetadata object MUST be exposed as the value of browser.ntpInfo

[Exposed=Window]
partial interface browser {
  readonly attribute NTPMetadata ntpInfo;
};

A UA MAY use a UA specific value in place of browser.

3.2. in CSS

The values available within NTPStyle can be very useful for developers to make a theme that is in keeping with the style choices of end users, and UAs. It is needlessly burdonsome to require javascript to use them, therefore UAs MAY expose the values as CSS Environment Variables 1 § 2 Environment Variables.

Any standard (i.e. non custom) property that are exposed in NTPStyle SHOULD also have a matching Environmental Variable Custom Property. For any exposed standard NTPStyle property, UAs MUST use the below listed <declaration-value>s as the names of the Environmental Variable Custom Property

Name Value
ntp-background-color <color>
ntp-background-image <image>
ntp-background-position <bg-position>
ntp-background-repeat <repeat-style>
ntp-item-hover-color <color>
ntp-link-color <color>
ntp-text-color <color>

For any exposed custom NTPStyle property (i.e. § 2.5 Extending properties), the <declaration-value> MUST follow the same naming requirements. Specifically,

Any new key MUST be prefixed by the letter "x" (U+0078 LATIN SMALL LETTER X), followed by the a dash (U+002D HYPHEN-MINUS).

x-ntp-custom-color <color>

For any exposed <declaration-value>, the stringified value exposed MUST be identical to the associated NTPStyle property value.

4. Permissions

This feature exposes a large amount of private data for the end user. Therefore it necessitates new key for the `permissions` field in WebExtension’s `manifest.json`. This key MUST be "newTabPageInfo" 
enum ManifestPermission {
  "newTabPageInfo"
};

Document conventions

Conformance requirements are expressed with a combination of descriptive assertions and RFC 2119 terminology. The key words “MUST”, “MUST NOT”, “REQUIRED”, “SHALL”, “SHALL NOT”, “SHOULD”, “SHOULD NOT”, “RECOMMENDED”, “MAY”, and “OPTIONAL” in the normative parts of this document are to be interpreted as described in RFC 2119. However, for readability, these words do not appear in all uppercase letters in this specification.

All of the text of this specification is normative except sections explicitly marked as non-normative, examples, and notes. [RFC2119]

Examples in this specification are introduced with the words “for example” or are set apart from the normative text with class="example", like this:

This is an example of an informative example.

Informative notes begin with the word “Note” and are set apart from the normative text with class="note", like this:

Note, this is an informative note.
